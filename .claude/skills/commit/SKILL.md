# commit: 변경사항 분석·컨벤션 커밋

변경사항을 읽고 논리적 단위로 묶어, `type: 요약` 형식의 한국어 커밋 메시지로 커밋한다.

## 절차

### 1. 변경사항 파악

병렬로 다음을 실행해 전체 상태를 본다.

```bash
git status
git diff            # unstaged 변경
git diff --staged   # 이미 스테이징된 변경
git log --oneline -10   # 기존 커밋 컨벤션·문체 확인
```

- untracked 파일이 있으면 내용도 확인한다 (`git status` 의 `??` 항목).
- **커밋하면 안 될 것이 섞여 있는지 먼저 점검**: `.env`, 비밀키, 토큰, 대용량 산출물, `node_modules`, 디버그용 임시 파일. 발견 시 커밋에서 제외하고 사용자에게 알린다.
- 기존 로그가 있으면 그 문체(이모지 사용 여부, prefix 표기)를 따른다. 로그가 비어 있으면 아래 기본 컨벤션을 쓴다.

### 2. 논리적 단위로 분류

- 서로 관련 없는 변경(예: 기능 추가 + 무관한 오타 수정)은 **별도 커밋으로 분리**한다.
- 한 기능을 이루는 변경(구현 + 그 테스트 + 관련 문서)은 **하나의 커밋**으로 묶는다.
- 분리가 필요하면 `git add <경로>` 로 파일/단위별로 스테이징한다. `-p`(hunk 단위)는 이 환경에서 인터랙티브 모드가 막혀 있으니 파일 단위로 나눈다.

### 3. 커밋 메시지 작성 (한국어)

형식: `type: 요약`

```
feat: 프롬프트 예시 칩 클릭 시 자동 입력 추가
```

**type**
- `feat`: 새 기능
- `fix`: 버그 수정
- `refactor`: 동작 변화 없는 구조 개선
- `chore`: 빌드·설정·의존성 등 기타
- `docs`: 문서만 변경
- `style`: 포매팅·세미콜론 등 (로직 변화 없음)
- `test`: 테스트 추가·수정

**요약 규칙**
- 한국어로, 명령형이 아니라 "무엇을 했는지" 간결히 (약 50자 이내).
- 마침표로 끝내지 않는다.
- 무엇을 왜 바꿨는지 본문이 필요하면 요약 아래 빈 줄 후 서술한다 (선택).

### 4. 승인 후 커밋

- 커밋 **전에** 제안 메시지(들)와 각 커밋에 포함될 파일 목록을 사용자에게 보여주고 승인을 받는다.
- 승인 후 실행. 여러 커밋이면 순서대로 스테이징→커밋을 반복한다.

```bash
git commit -m "feat: ..."
```

- 본 리포지토리가 `main`/기본 브랜치에 있고 사용자가 브랜치를 지정하지 않았다면, 커밋 전에 브랜치 분기 여부를 확인한다.
- **푸시는 하지 않는다.** 사용자가 명시적으로 요청할 때만 push 한다.
- 커밋 메시지 끝에는 환경에서 요구하는 Co-Authored-By / Claude-Session 푸터를 붙인다(해당 환경 지침이 있을 때).

## 인자

- `--all`: unstaged·untracked 포함 전체를 대상으로 분석한다 (기본은 전체 확인 후 분류 제안).
- `--no-verify`: pre-commit 훅을 건너뛴다 (`git commit --no-verify`). 훅 실패로 막힐 때만 사용하고, 사용자에게 이유를 알린다.

## 하지 말 것

- 사용자 승인 없이 커밋하지 않는다 (`--auto` 류 옵션 없음).
- `git add -A` 로 무분별하게 전부 담지 않는다. 검토한 파일만 담는다.
- `.env` 등 민감 파일을 커밋하지 않는다.
- 요청하지 않은 push·force-push·rebase·amend 를 하지 않는다.
