# TDD Rule

> **이 규칙은 Rigid다. 상황에 맞게 변형하지 마라.**
> 아래 절차는 재량이 아니라 강제다. "이번만 예외"는 없다. 예외가 필요하면 규칙을 어기는 게 아니라 아래 **프로젝트별 규칙 우선 조항**을 따른다.

---

## 1. 적용 기준 구분

### TDD를 **반드시** 적용하는 대상

- **비즈니스 로직** — 도메인 규칙, 계산, 상태 전이, 검증
- **API / 엔드포인트** — 요청·응답 계약, 라우팅, 에러 매핑
- **유틸 / 순수 함수** — 파서, 포매터, 변환기, 헬퍼
- **버그 수정** — 재현하는 실패 테스트를 **먼저** 쓰고 고친다 (회귀 방지)
- 데이터 변환, 조건 분기, 경계값이 있는 모든 코드

### TDD가 **불필요**한 대상

- **타입 정의** (인터페이스, 타입 별칭, enum)
- **설정 파일** (config, env 스키마, 빌드 설정)
- **순수 UI / 마크업** (로직 없는 프레젠테이션 컴포넌트, 스타일)
- **SQL / 마이그레이션** (선언적 스키마)
- 상수 선언, 단순 재노출(re-export), 자동 생성 코드

> 판단 기준: **"틀릴 수 있는 동작(behavior)이 있는가?"** 있으면 적용, 없으면 불필요.
> 애매하면 적용한다.

---

## 2. RED-GREEN-REFACTOR 사이클

### RED — 실패하는 테스트를 먼저 쓴다

- **하나의 동작 = 하나의 테스트.** 한 테스트에 여러 동작을 담지 마라.
- 테스트를 **반드시 실행해서 실패를 눈으로 확인**한다. 실행하지 않은 RED는 RED가 아니다.
- 실패 이유가 **"기능 미구현"**이어야 한다. 오타·import 오류·컴파일 에러로 인한 실패는 무효 — 고치고 다시 실패시켜라.
- 테스트는 구현이 아니라 **원하는 동작**을 서술한다.

### GREEN — 통과할 **최소한**의 코드만 쓴다

- 테스트를 통과시키는 **가장 단순한 코드**만 작성한다. 하드코딩이라도 통과가 먼저다.
- **YAGNI.** 지금 테스트가 요구하지 않는 기능·분기·추상화·범용성을 넣지 마라.
- **신규 테스트 + 기존 테스트 전부** 통과를 확인한다. 하나라도 깨지면 GREEN이 아니다.

### REFACTOR — green을 유지한 채 정리한다

- **중복 제거 · 이름 개선 · 헬퍼 추출**만 한다.
- 리팩터 중 테스트는 **계속 green**이어야 한다. 매 단계 실행해서 확인한다.
- **새 동작을 추가하지 마라.** 새 동작은 다음 RED에서.

### 반복

- 다음 동작에 대한 **RED로 돌아간다.** 한 번에 하나의 동작씩 사이클을 돈다.

---

## 3. 삭제 강제 규칙

- 테스트보다 **프로덕션 코드를 먼저 작성했다면**, 그 코드를 **삭제하고 RED부터 다시 시작**한다.
- **"참고용"으로 주석 처리하거나 남겨두는 것도 금지.** 지운다.
- 이유: 구현을 먼저 보면 테스트가 구현에 맞춰 편향된다. TDD의 설계 이점이 사라진다.
- 삭제가 아까우면 그건 TDD를 건너뛰었다는 증거다. 그대로 삭제한다.

---

## 4. 변명 차단표

| 변명 | 반론 |
|------|------|
| "너무 단순해서 테스트 불필요" | 단순하면 테스트도 1줄이다. 단순함은 생략의 근거가 아니라 **비용이 없다는 근거**다. 단순한 코드가 조용히 깨진다. |
| "나중에 추가하겠다" | 나중은 오지 않는다. 그린 상태로 넘어가면 되돌아오지 않는다. **지금의 RED가 유일한 기회**다. |
| "시간이 없다" | 테스트 없는 코드는 디버깅·회귀로 **더 많은 시간을 쓴다.** TDD는 시간을 쓰는 게 아니라 앞당겨 절약한다. |
| "삭제하면 낭비다" | 편향된 테스트를 유도하는 코드가 더 큰 낭비다. 이미 이해한 내용은 **RED→GREEN에서 몇 분이면 복원**된다. |
| "프로토타입이라 괜찮다" | 프로토타입은 살아남아 프로덕션이 된다. 버릴 코드라면 테스트도 필요 없을 만큼 **정말 버려라.** 남길 거면 테스트한다. |

---

## 5. 프로젝트별 규칙 우선 조항

- 프로젝트의 **`CLAUDE.md` / `AGENTS.md`에 TDD 관련 규칙이 있으면 그것을 우선**한다 (테스트 러너, 커버리지 기준, 파일 배치, 예외 범위 등).
- **이 파일은 기본값(fallback)이다.** 프로젝트가 명시하지 않은 부분에만 적용된다.
- 프로젝트 규칙과 이 파일이 충돌하면 **프로젝트 규칙이 이긴다.** 단, 프로젝트 규칙이 "TDD를 하지 않는다"고 명시하지 않는 한, 위 RED-GREEN-REFACTOR 절차는 유지한다.
